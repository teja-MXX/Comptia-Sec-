## 1. What is a Digital Certificate?

A digital certificate is a digitally-signed electronic document that binds a public key to an identity.

### A ‚Äúuser‚Äù can be:

- A human
    
- A server
    
- A workstation
    
- A device/application
    

### Certificates follow:

- X.509 standard (defines structure ‚Üí subject info, issuer info, public keys, signatures)
    

### A certificate contains:

- Subject‚Äôs name (person/server/organization)
    
- Subject‚Äôs public key
    
- Issuer (Certificate Authority) information
    
- Validity period (start/end)
    
- Serial number
    
- CA's digital signature

# 1Ô∏è‚É£ Wildcard Certificates

Wildcard certificate = One certificate that covers all subdomains under the same root domain.

Example:  
Wildcard for `*.example.com` covers:

- `www.example.com`
    
- `courses.example.com`
    
- `api.example.com`
    

### Pros:

- Cheaper ‚Üí one certificate covers many subdomains
    
- Easier to manage ‚Üí only one cert to renew
    
- Widely used in multi-service architectures
    

### Cons:

- If compromised ‚Üí all subdomains become compromised
    
- Revocation affects all servers
    
- Must reissue and redistribute everywhere

# 2Ô∏è‚É£ Single-Sided vs Dual-Sided Certificates

### Single-Sided Certificate

- Only the server presents a certificate to the client.
    
- Example: normal HTTPS websites.
    
- Browser verifies server ‚Üí establishes TLS.
    

### Dual-Sided Certificate (Mutual TLS / mTLS)

- Server verifies client certificate
    
- Client verifies server certificate
    
- Used only in high-security systems
    
- Requires more processing overhead

# 3Ô∏è‚É£ Self-Signed Certificates

A self-signed certificate is signed by the entity itself, not a CA.

### Properties:

- Provides encryption, but NOT trust
    
- Browsers show warnings
    
- Good for testing / internal use
    
- Bad for public websites (no third-party validation)

# 4Ô∏è‚É£ Third-Party Certificates

Certificates issued by a trusted Certificate Authority (CA) such as:

- VeriSign
    
- DigiCert
    
- GlobalSign
    
- AWS, Google, Cloudflare CAs
    

### Why trusted?

Operating systems and browsers embed CA root certificates.

This means:  
‚Üí Any certificate issued by those CAs is automatically trusted.

# 5Ô∏è‚É£ Root of Trust (Chain of Trust)

A certificate is trusted because:

- It is signed by another trusted certificate (intermediate CA)
    
- Which is signed by another trusted certificate
    
- ‚Ä¶ leading up to the Root CA
    

### Chain:

`End-entity certificate (your website cert) ‚≠° Signed by Intermediate CA ‚≠° Signed by Root CA (most trusted)`

If the Root CA is compromised, every certificate below it becomes compromised.

# 6Ô∏è‚É£ Certificate Authority (CA)

A Certificate Authority:

- Issues digital certificates
    
- Signs them
    
- Stores public key copies
    
- Maintains certificate revocation lists
    
- Verifies identity before issuing
    

Certificate contains:

- CA name
    
- CA digital signature
    
- Serial number
    
- Validity dates
    
- Version number

# 7Ô∏è‚É£ Registration Authority (RA)

An RA is like a ‚Äúfront desk‚Äù for a CA.

### Responsibilities:

- Accept certificate requests
    
- Verify the user‚Äôs identity
    
- Pass validated requests to the CA
    

CA signs the certificate; RA handles user verification.

# 8Ô∏è‚É£ Certificate Signing Request (CSR)

A CSR is a block of encoded text generated when requesting a certificate.

### Contains:

- The public key
    
- Organization name
    
- Domain name
    
- Location
    
- Email
    
- Country
    

### Important:

The private key NEVER leaves the requester‚Äôs machine.  
CA only receives:

- Public key
    
- Metadata
    

CA validates ‚Üí signs ‚Üí returns certificate.

# 9Ô∏è‚É£ Certificate Revocation List (CRL)

A CRL is a full list of all revoked certificates issued by a CA.

Reasons for revocation:

- Key compromise
    
- Mis-issuance
    
- Owner leaves company
    
- Malicious activity
    

When your browser requests a certificate:

- CA checks CRL ‚Üí if certificate is revoked, browser will reject connection.

# üîü OCSP (Online Certificate Status Protocol)

OCSP checks one certificate‚Äôs revocation status using its serial number.

### Properties:

- Faster than CRL
    
- Does NOT use encryption ‚Üí less secure
    
- Returns:
    
    - GOOD
        
    - REVOKED
        
    - UNKNOWN

# 1Ô∏è‚É£1Ô∏è‚É£ OCSP Stapling

Optimization of OCSP.

Instead of the client querying the OCSP server:

- The web server itself periodically fetches its OCSP status
    
- ‚ÄúStaples‚Äù (attaches) it to the TLS handshake
    

### Benefits:

- Faster TLS handshake
    
- Less load on the CA
    
- No privacy leak (CA can‚Äôt track who visited which site)

# 1Ô∏è‚É£2Ô∏è‚É£ Public Key Pinning

Used to prevent certificate impersonation attacks.

### Idea:

Server sends a list of trusted public keys in the HTTP header.

Browser checks:

- ‚ÄúDoes the certificate I received match one of the pinned keys?‚Äù
    

If NOT:  
‚Üí Browser warns of attack ‚Üí blocks connection

Note: Public key pinning is deprecated in browsers due to risk of locking out websites if misconfigured.

# 1Ô∏è‚É£3Ô∏è‚É£ Key Escrow Agents

Key escrow = storing a secure backup copy of private keys.

Used when:

- User loses private key
    
- Employee leaves company
    
- Legal access required
    

Handled by key escrow agent (person/system).

Security measures:

- Must protect escrow store
    
- Require two administrators to retrieve keys (separation of duties)
    

---

# 1Ô∏è‚É£4Ô∏è‚É£ Key Recovery Agents

These are software systems that recover:

keys.

Used for disaster recovery of CA keys.

If the CA is destroyed (fire, ransomware, etc.), the key recovery agent can restore keys.
